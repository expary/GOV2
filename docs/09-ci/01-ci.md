# 09.01 CI

GOV2 uses GitHub Actions to run the same validation command that developers run locally.

## Workflow

The workflow lives at:

- [`.github/workflows/ci.yml`](../../.github/workflows/ci.yml)

It runs on:

- pull requests
- pushes to `main`
- pushes to `master`

## Validation Steps

CI performs:

1. checkout
2. Go setup from `go.mod`
3. Node setup for the Vue frontend
4. `npm ci --prefix web`
5. `make validate`
6. `git diff --exit-code`

`make validate` includes:

- `go mod tidy`
- `go test ./...`, including OpenAPI route-summary alignment, path/query parameter type alignment against backend route metadata, `400 BadRequestError` or `400 ValidationError` response alignment against backend route metadata, module registry validation for uniqueness, route path/method shape, and metadata consistency, and built-in module backend route metadata alignment
- `cd web && npm run api:check`, including generated endpoint metadata and request wrapper JSDoc with OpenAPI operation summaries, request-body standard `400` response component checks, named request wrapper import and AbortSignal option docs, view-level GET request AbortSignal usage, object/array request-response schema type, non-text success response schema coverage, and permission freshness plus frontend API and permission usage checks
- `cd web && npm run test`, covering API client runtime contract behavior including public endpoint auth suppression, generated response schema validation, and AbortSignal forwarding
- `cd web && npm run lint`
- `cd web && npm run build`, covering compiled Vue views that pass AbortSignals through generated API requests for superseded dashboard, module metadata, and core system list refreshes
- `node scripts/check-doc-links.mjs` for local Markdown links and heading anchors in `README.md`, `AGENTS.md`, and `docs/`, plus numbered design-document placement and document-map/docs README coverage

The final diff check fails CI when `go mod tidy`, generated frontend API files,
generated permission metadata, or other tracked validation outputs are stale and
need to be committed.

Release packaging can be checked locally with:

```bash
make package
bin/gov2 version
```

## PostgreSQL Integration Tests

CI also runs a PostgreSQL-backed integration job. It starts a `postgres:16-alpine` service container and runs:

```bash
make test-postgres
```

That command runs the SQL store integration test and the SQL-backed HTTP API integration test serially against the service database.

The job uses:

- database: `gov2`
- user: `gov2`
- password: `gov2`
- DSN: `postgres://gov2:gov2@localhost:5432/gov2?sslmode=disable`

Normal `make test` still skips PostgreSQL integration tests when `GOV2_TEST_POSTGRES_DSN` is absent.
